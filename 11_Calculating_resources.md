### Расчёт потребности в ресурсах

### NGINX Ingress

Пиковая нагрузка: 643 489 RPS (из раздела 4). Доля новых TLS-соединений - 20% → 128 698 CPS.

```
Ядер на TLS: 128 698 / 428 = 301 vCPU
Ядер на throughput: 128.7 Гбит/с / (0.367 Гбит/с·vCPU) = 351 vCPU  ← лимитирует
```

Выбираем ноды 16 vCPU / 32 ГБ / 10 Гбит/с:

```
351 / 16 = 21.9 → 22 рабочих ноды + 1 резерв = 23 ноды
```

### PostgreSQL + Citus (user-centric)

Нагрузка (из раздела 5): чтение ~38 750 avg RPS (с кешированием профилей в Redis большинство не доходит до БД - реальная нагрузка ~5-10%):

```
Эффективный RPS: 38 750 × 0.05 = ~1 938 RPS
Ядер: 1 938 / 500 = 3.9 → 4 ядра (нагрузка невысокая)
```

32 шарда × (1 primary + 2 replica) = 96 экземпляров, но они мультиплексированы на физических нодах. Выбираем ноды 16 cores / 64 ГБ / 2×2 ТБ NVMe:

Объём данных: users (~250 байт × 620 млн) + follows (~24 байт × 50 млрд связей) ≈ 1.4 ТБ. С индексами × 2 = ~3 ТБ.

```
3 ТБ × RF=3 / (4 ТБ на ноду) = 2.25 → 3 ноды минимум
Берём 6 нод (по 2 в зоне доступности), primary + replica на разных нодах
```

Нагрузка (самая тяжёлая таблица - tweet_views): 387 500 avg QPS записи + 387 500 avg QPS чтения = 775 000 ops/s avg, пик × 1.5 = 1 162 500 ops/s.

```
Ядер: 1 162 500 / 50 000 = 23.3 → 24 ядра
```

Объём данных за год (основные таблицы):

|Таблица|Суточная запись|За год|

|tweets|~465 ГБ|~170 ТБ|
|likes|~242 ГБ|~88 ТБ|

Нода ScyllaDB: 32 cores / 128 ГБ / 4×4 ТБ NVMe = 16 ТБ:

```
825 ТБ / 16 ТБ = 51.6 → 52 ноды
```

Проверка по CPU: 52 × 32 = 1 664 ядра, ops = 1 664 × 50 000 = 83.2 млн ops/s - запас более чем достаточный.

```
Берём 54 ноды (кратно 3 зонам доступности)
```

```
Ядер: 826 667 / 100 000 = 8.3 → 9 ядер

```

### Qdrant

Нагрузка: ~12 917 avg RPS чтения. Объём: 620 млн × 1 КБ = 620 ГБ.

```
Ядер: 12 917 / 1 000 = 12.9 → 13 ядер  (ANN-поиск дороже простого чтения)
Нода: 16 cores / 128 ГБ / 2×1 ТБ NVMe
620 ГБ × RF=3 / 128 ГБ = 14.5 → 3 ноды (по одной в зоне доступности)
```

### MinIO S3

Пиковая нагрузка записи (upload): 26.96 ТБ/сутки = ~2.5 Гбит/с avg, пик × 4 = 10 Гбит/с. Пиковая нагрузка чтения (из раздела 2): 9.37 Тбит/с - полностью снимается CDN, до MinIO доходят только cache miss (~1-5%).

```
Эффективный пиковый RPS чтения MinIO: 9.37 Тбит/с × 0.02 = 187 Гбит/с
```

Объём хранения (медиа, 3 года): 0.81 ПБ/мес × 36 = 29.2 ПБ.

Нода MinIO: 12 cores / 32 ГБ / 8×16 ТБ HDD = 128 ТБ. Erasure coding 8+4 → полезная ёмкость 2/3:

```
29.2 ПБ / (128 ТБ × 2/3) = 29 200 ТБ / 85.3 ТБ = 342 → 342 ноды
```

Сеть на ноду: 187 Гбит/с / 342 = ~550 Мбит/с - укладывается в 2×10 Гбит/с.
